# ADR-003: ファイルID管理によるメモリ効率化戦略

## ステータス

**現在のステータス:** 提案中

**決定日:** 2025-06-15

**関連する決定:**

- 関連する決定: [ADR-001 技術スタック選定](/docs/adr/ADR-001-technology-stack-selection.md)
- この決定により置き換えられる: なし

## コンテキスト

### 背景・状況

i18n Language Serverは、プロジェクト内のすべてのJavaScript/TypeScriptファイルを解析し、翻訳キーの使用箇所（Reference）を追跡する必要があります。大規模プロジェクトでは、ファイル数が数千〜数万に及び、各ファイルパスの文字列長も長くなる傾向があります。

**現在の状況:**

- 大規模プロジェクトでは数千〜数万のファイルを扱う必要がある
- 各ファイルパスは平均50-100文字程度
- 翻訳キーの参照情報は各ファイルパスを含む必要がある
- メモリ効率的な設計が求められる

**解決すべき問題:**

- ファイルパス文字列の重複保存によるメモリ消費
- 参照情報のサイズ増大によるパフォーマンス低下
- 大規模プロジェクトでの実用性確保

### 制約条件

**技術的制約:**

- Rustの所有権システムに適合する設計
- 並行処理に対応（DashMap等の使用）
- LSPプロトコルとの互換性維持

**パフォーマンス制約:**

- メモリ使用量：1000キーで50MB以下
- ファイルパス解決のオーバーヘッド最小化
- リアルタイム処理に支障なし

---

## 検討した選択肢

### 選択肢A: ファイルパスを直接保存

**概要:**
各参照情報に完全なファイルパスを文字列として保存する。

**メリット:**

- 実装がシンプル
- ファイルパスの解決が不要
- デバッグが容易

**デメリット:**

- メモリ使用量が大きい（パス文字列の重複）
- 参照情報のサイズが可変長
- キャッシュ効率が悪い

**実装コスト:** 低  
**技術的複雑さ:** 低  
**リスク:** 高（メモリ不足）

### 選択肢B: ファイルパスのハッシュ値使用

**概要:**
ファイルパスをハッシュ化し、64ビット整数として保存。

**メリット:**

- 固定サイズ（8バイト）
- メモリ使用量の削減
- 高速な比較

**デメリット:**

- ハッシュの衝突リスク
- 元のパス復元に別途マッピングが必要
- デバッグが困難

**実装コスト:** 中  
**技術的複雑さ:** 中  
**リスク:** 中（ハッシュ衝突）

### 選択肢C: インクリメンタルID管理

**概要:**
ファイルパスに連番のIDを割り当て、双方向マッピングで管理。

**メリット:**

- 最小サイズ（4バイト）
- 衝突リスクなし
- 明確なファイル識別

**デメリット:**

- ID管理の実装が必要
- 双方向マッピングの保守
- 起動時の初期化処理

**実装コスト:** 中  
**技術的複雑さ:** 中  
**リスク:** 低

### 比較表

| 観点               | 選択肢A | 選択肢B | 選択肢C | 重要度 |
| ------------------ | ------- | ------- | ------- | ------ |
| **メモリ効率**     | 低      | 高      | 最高    | 高     |
| **実装複雑度**     | 低      | 中      | 中      | 中     |
| **安全性**         | 高      | 中      | 高      | 高     |
| **パフォーマンス** | 中      | 高      | 高      | 高     |
| **保守性**         | 高      | 中      | 高      | 中     |

## 決定

### 採用する選択肢

**選択:** 選択肢C（インクリメンタルID管理）

**決定理由:**

1. **最高のメモリ効率**: 32ビット整数（4バイト）で表現でき、大規模プロジェクトでも実用的
2. **安全性**: ハッシュ衝突のリスクがなく、確実なファイル識別が可能
3. **拡張性**: 将来的な機能追加（ファイル属性管理等）に対応しやすい

### 決定の詳細

**実装方針:**

```rust
// ファイルIDの型定義
type FileId = u32;

// ファイルID管理構造体
struct FileIdManager {
    path_to_id: DashMap<PathBuf, FileId>,
    id_to_path: DashMap<FileId, PathBuf>,
    next_id: AtomicU32,
}

// 参照情報の構造体
struct TranslationKeyReference {
    file_id: FileId,      // 4 bytes (PathBufの代わり)
    line: u32,            // 4 bytes
    column: u32,          // 4 bytes
    // 合計: 12 bytes + α
}
```

**設計原則:**

- FileIdは再利用しない（削除されたファイルのIDも保持）
- 双方向マッピングで高速な相互変換を保証
- 並行処理対応（DashMapとAtomicU32）

## 結果・影響

### ポジティブな影響

**短期的効果:**

- メモリ使用量を約80%削減（パス文字列50文字→4バイト）
- 参照情報の固定サイズ化によるキャッシュ効率向上
- 比較処理の高速化（整数比較）

**長期的効果:**

- 大規模プロジェクトでの実用性確保
- ファイル属性（権限、タイプ等）の効率的な管理基盤
- メモリ局所性の向上によるパフォーマンス改善

### ネガティブな影響・トレードオフ

**受け入れるコスト:**

- FileIdManager実装の追加複雑度
- 起動時のID割り当て処理時間
- デバッグ時のID→パス変換の手間

**リスク・課題:**

- FileIdの枯渇（u32の上限：約42億ファイル）
- 軽減策：実用上問題ないレベル、必要に応じてu64に拡張可能
